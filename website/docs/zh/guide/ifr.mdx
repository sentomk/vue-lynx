# 首屏直出（IFR）

首屏直出（Instant First-Frame Rendering，IFR）让页面在加载完成的那一刻就显示真实内容——没有白屏，也没有中间加载态。它是 Lynx 官方 React 框架自带的同名优化在 Vue Lynx 上的移植。平台层面的背景介绍见 [Lynx IFR 指南](https://lynxjs.org/guide/interaction/ifr)。

## 两个开关

Vue Lynx 的首帧性能由两个相互独立、可自由组合的插件选项控制：

| 开关 | 改变的是什么 | 机制 |
|---|---|---|
| `enableIFR` | 首帧**何时**绘制：在 `loadTemplate` 期间、任何后台 JavaScript 运行之前——而不是等后台线程启动 + 渲染 + IPC 之后。 | 主线程 bundle 携带 Vue 运行时和你的应用；首屏在主线程上渲染，后台渲染完成 hydration。 |
| `enableElementTemplates` | 渲染**花多少钱**：结构静态的子树在编译期降为原生元素模板，把逐节点的 vdom + ops + 解释器成本折叠成每棵子树一次操作。 | 编译期模板下沉——对应 ReactLynx snapshot 体系的 Vue 实现。深入解读见[元素模板](/zh/guide/element-templates)。 |

二者可组合：元素模板同时加速普通管线、更新路径和 IFR 主线程渲染（下沉后的 ops 流走同一套 hydration 协议）。内容型首屏建议两个都开；[下方的 benchmark](#实测性能) 量化了各自的收益。

## 白屏从哪来

在 Vue Lynx 默认的双线程管线中，所有可见内容都由后台线程产出：

```
  没有 IFR 时：

  ┌───────────────┐   ┌─────────────────────┐   ┌───────────────┐
  │    主线程     │   │      后台线程       │   │    主线程     │
  │  （空页面）   │──▶│（Vue 渲染并发送     │──▶│（应用 ops，   │
  │               │   │  ops 批次）         │   │  完成首帧）   │
  └───────────────┘   └─────────────────────┘   └───────────────┘
        ▲                                               │
        └────────── 用户盯着一帧白屏 ───────────────────┘
```

主线程在 `loadTemplate` 期间只创建了一个空页面，然后开始等待：后台线程要启动、执行应用 bundle、跑完 Vue 的首次渲染，再把产生的 ops 跨线程发回来，屏幕上才会出现内容。

## Vue Lynx 中的 IFR 工作原理

启用 IFR 后，主线程 bundle 会携带完整的 Vue 运行时和你的应用代码。首屏在 `loadTemplate` 期间、**在主线程上**同步渲染完成：

```
  启用 IFR 后：

  ┌────────────────────────────────────────────────┐
  │                    主线程                      │
  │  loadTemplate ──▶ Vue 渲染 ──▶ 直接绘制首帧    │
  └────────────────────────────────────────────────┘
                          │（并行进行）
  ┌────────────────────────────────────────────────┐
  │                   后台线程                     │
  │  启动 ──▶ 渲染同一应用 ──▶ hydration           │
  └────────────────────────────────────────────────┘
```

1. **主线程直出首帧。** Lynx 调用 `renderPage` 时，Vue 应用直接在主线程上挂载，渲染产生的元素操作通过 Element PAPI 立即执行——首帧在任何后台 JavaScript 运行之前就已上屏。每个已应用的 ops 批次同时被记录下来。
2. **后台渲染。** 后台线程启动后用同样的数据渲染同一个应用。由于两个线程执行完全相同的确定性代码，它产生的元素 id 和事件 sign 与主线程渲染逐一对齐。
3. **Hydration。** 后台线程的初始 ops 批次会与记录的 ops 流做比对，而不是再次应用：
   - 完全一致的批次会被**跳过**——内容已经在屏幕上；
   - 值级差异（文本、样式、属性）会被**原地打补丁**；
   - 结构性差异则拆除首屏树、按后台 ops 重建——正确性从不依赖两次渲染一致，只有性能收益依赖。

Hydration 完成后，树的所有权交给后台线程，后续更新都走正常的 ops 管线。首帧期间绑定的事件使用与后台线程一致的确定性 sign，点击无需任何重新绑定即可路由到你的 Vue 事件处理函数。

### 设计原理

为什么在主线程上**重放完整运行时**，而不是发一份预渲染的快照？因为确定性把整套 hydration 协议白送了：两个线程跑同样的代码，主线程记录的 ops 流与后台线程的初始批次逐帧对齐，hydration 退化为一次廉价的比对遍历——不需要 id 重映射、不需要独立的快照格式、不需要维护第二条代码路径。代价是完整重放做的 JS 工作量与普通渲染相同：它搬运了工作，而不是消减工作。这正是 `enableElementTemplates` 补上的缺口：静态骨架——通常占首屏节点数的大头——在两个线程上都不再支付 vdom、ops 和解释器成本。在收敛到这套设计之前，我们把备选路线（直接 PAPI 应用、纯静态岛、Vapor 式无 vdom 渲染器）都做了原型并测量，完整的阶梯数据见[策略基准报告](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/REPORT.md)。

## 启用方式

在插件选项中开启：

```ts title="lynx.config.ts"
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      enableIFR: true,
      enableElementTemplates: true,
    }),
  ],
})
```

典型的组件树无需修改任何应用代码。

## 实测性能

以下数字来自三轮测量，完整数据与方法学在仓库里（[策略基准](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/REPORT.md)、[全示例扫描](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/EXAMPLES-REPORT.md)、[真实浏览器验证](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/VERIFICATION.md)）。

### 真实浏览器、真实线程（Lynx for Web）

测 IFR 最诚实的方式是有真实线程边界的环境。在 Lynx for Web 上（后台运行时跑在 Web Worker 里、真实 postMessage IPC、headless Chromium、7 次取中位）：

| bundle | FCP 无 IFR | FCP 有 IFR | Δ |
|---|---|---|---|
| hello-world | 92.0 ms | 74.3 ms | **−19%** |
| gallery（304 节点） | 127.9 ms | 106.2 ms | **−17%** |
| ReactLynx 探针（85 节点，对照组） | 99.2 ms | 68.9 ms | **−31%** |

4× CPU 节流（移动端代理）下元素模板反超纯 IFR：hello-world FCP −17%（`ifr+et`）vs −12%（`ifr`）。收益是结构性的——IFR 把后台线程启动 + IPC 从关键路径上移除——与 ReactLynx IFR 在同一 harness 上表现出的收益属同一类。

### 渲染的 JS 成本（策略阶梯）

约 1000 节点的合成首屏，`--jitless` V8（最接近 Lynx 主线程解释器的代理），动态内容场景 warm 中位：

| 管线 | 渲染耗时 |
|---|---|
| 经典后台渲染 | 9.4 ms |
| IFR 重放（`enableIFR`） | 8.4 ms |
| + 元素模板（`ifr+et`） | **1.3 ms（7×）** |
| 直线 PAPI 地板 | 0.4 ms |

元素模板还把跨线程 ops 载荷缩小 3–1000×（取决于屏幕静态程度：一个 1400 节点的静态屏从 78 KB 缩到 69 字节）。

### 代价（24 个真实示例）

| 指标 | 中位影响 |
|---|---|
| `main.lynx.bundle` 体积（gzip） | **×2.26**——主线程要多带一份 Vue 运行时 + 应用，这是最主要的成本 |
| 可交互前的串行 JS 工作量（TTI 代理） | ×1.36 上界——应用在两个线程各求值一次；真机上与后台启动重叠 |
| 最终渲染文档 | 22/23 个可构建示例在 off / ifr / ifr+et 三配置下逐字节一致（属性顺序无关比较） |

### 该不该开？

| 首屏画像 | 建议 |
|---|---|
| 静态/内容优先，对体积不敏感 | **开，两个都开**——首帧在 `loadTemplate` 完整绘制 |
| 数据驱动但初始数据同步可得 | 开 |
| 完全依赖网络请求的首屏 | 不开——付出体积却没有 FCP 收益（web 上最大的此类示例实测 FCP +19%）；或者重构出一个真正的骨架屏 |
| 体积极度敏感 | 先测——目前 gzip 大约翻倍 |

## 编写 IFR 友好的首屏

首屏渲染会执行两次——每个线程一次——因此它必须表现得像输入的纯函数：

- **保持渲染输出确定性。** 首屏渲染路径上避免 `Math.random()`、`Date.now()` 或依赖线程的分支。如果两次渲染出现结构性分歧，Vue Lynx 会检测到并回退为完整的后台渲染——页面依然正确，但该屏失去 IFR 收益（dev 构建会输出警告）。
- **副作用放进生命周期钩子。** `onMounted`、`onUnmounted` 等钩子在主线程渲染期间不会执行（与 ReactLynx 中 `useEffect` 不在主线程触发一致）。数据请求、定时器、订阅只会在后台线程运行一次。注意：目前只有组合式 API 的钩子被抑制——Options API 的 `mounted()` 仍会在主线程渲染期间触发，首屏组件请优先使用组合式 API 的生命周期钩子。
- **首屏只画同步数据。** IFR 绘制的是初始数据能渲染出的内容。依赖网络请求的内容仍要等后台线程取回后出现——可以为这些区域准备骨架屏，让 IFR 消除它们周围的白屏。
- **后台线程就绪前的事件会被丢弃。** 从首帧绘制到后台线程完成启动之间有一个很短的窗口（通常几毫秒），常规事件处理函数在那之后才生效。[主线程脚本](/zh/guide/main-thread-script)事件不受影响——它们本来就在主线程上。

## 取舍

- 主线程 bundle 大约增加一个 Vue 运行时加应用代码的体积（在我们的示例集上 gzip 中位 ×2.26），模板变大、`loadTemplate` 的解析时间变长。原生 Lynx 的主线程脚本以预编译字节码分发，解析项很小；Lynx for Web 上它落在 FCP 路径上——这正是大 bundle + 纯骨架首屏在 web 上可能反转的原因。对内容型首屏来说，感知延迟的收益通常占优；建议按自己的应用实测。
- 当 bundle 本身需要异步加载，或整个首屏都依赖异步数据时，IFR 无能为力——与 ReactLynx IFR 的约束相同。
- CSS Modules（`<style module>`）的 class 名首帧渲染为未哈希版本，hydration 时补齐；主线程完整支持 CSS Modules 是已跟踪的后续工作。

## 试一试

仓库里的每个示例都已开启两个开关：

```bash
git clone https://github.com/Huxpro/vue-lynx
cd vue-lynx && pnpm install && pnpm build
cd examples/hello-world && pnpm build   # 或任何其他示例
```

每次构建在 `dist/` 下产出两个目标：

- **`main.lynx.bundle`**——在 LynxExplorer 中加载：`lepusCode` 段现在包含了你的应用，完整首屏会在 `background.js` 执行任何一行代码之前绘制出来。
- **`main.web.bundle`**——用 [Lynx for Web](https://lynxjs.org/guide/start/fragments/web/platform-features) 的 `<lynx-view>` 加载，可在浏览器中观察同样的行为（后台运行时跑在真实的 Web Worker 里）。
