# Element Templates

Element templates are Vue Lynx's compile-time template lowering — the Vue
counterpart of ReactLynx's snapshot system. Eligible template subtrees are
compiled into native element templates: the static skeleton becomes a
straight-line element-creation function executed on the main thread in a
single operation, and interior dynamic parts ("holes") are updated through
the ordinary ops protocol.

## Why

In the default pipeline, every element — however static — costs a vnode, a
shadow node, ~5 ops frames across the thread boundary, and an interpreter
dispatch on the main thread. For typical first screens the static scaffolding
dominates the node count, so this per-node cost dominates first-frame JS
time. Lowering removes it:

```
  Without element templates (per static node):
  vnode → ShadowElement → 5 ops frames → JSON → interpreter → PAPI

  With element templates (per lowered subtree):
  one vnode → one INSTANTIATE_TEMPLATE op → one straight-line create()
```

In our [strategy benchmark](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/REPORT.md),
block-level lowering rendered typical ~1000-element first screens **7–15×
faster** than the unlowered pipeline under an interpreter (the closest proxy
for Lynx's main-thread engine) and shrank the cross-thread ops payload by
**3–1000×** — a win that applies to the normal pipeline, to updates, and to
[IFR](/guide/ifr) alike (they compose: lowered ops streams hydrate through
the same fast path).

## Enabling

```ts title="lynx.config.ts"
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      enableElementTemplates: true,
    }),
  ],
})
```

No app-code changes are required, and no template syntax is affected.

## What gets lowered

A subtree is lowered when its **structure** is compile-time-known: every
node is a plain element, and only prop values / text content are dynamic.

```vue
<view class="card">                    <!-- lowered: one template          -->
  <image class="icon" src="a.png" />   <!--   baked into the skeleton      -->
  <text class="title">{{ title }}</text><!--  '#text' hole                 -->
  <view :class="badgeCls">…</view>     <!--   'class' hole                 -->
</view>
```

Structural features always stay on the normal vdom path — correctness never
depends on lowering:

- components, `<slot>`, `v-if` / `v-for` hosts (their *bodies* can still be
  templates — v-for items typically lower to one instantiation per item)
- elements with runtime directives (`v-show`, `v-model`, custom), `ref`,
  `key`, `id`, or vnode hooks — except on the subtree **root**, which keeps
  all of its props and directives on the vnode and behaves exactly like a
  normal element (Transition on a lowered root works)
- `<list>` / `<list-item>` (they have dedicated main-thread handling)
- comments and mixed dynamic text runs

Scoped CSS is fully supported — the component's CSS scope id is baked into
the skeleton at compile time.

## Relationship to Lynx's native element templates

The Lynx engine has a native mechanism of the same name: element trees can be
encoded into a binary `elementTemplates` section of the bundle and
instantiated by the engine directly via the `__ElementFromBinary` PAPI, with
named parts resolved through `__GetTemplateParts` — no JavaScript involved.

**Vue Lynx's element templates do not use that mechanism today.** The
"template" here is a compiler-generated JavaScript `create()` function of
ordinary element PAPI calls, registered in a framework-level registry and
triggered by Vue Lynx's own `INSTANTIATE_TEMPLATE` op. To the engine it looks
like a normal sequence of element creations — what the lowering removes is
the framework work *around* those calls (vdom, per-node ops, interpreter
dispatch), which is where the measured wins come from. ReactLynx's snapshots
make the same choice: they too emit straight-line PAPI calls rather than
binary templates.

The two designs are complementary rather than competing: the compiler output
(static skeleton + hole list) is exactly what a binary encoding needs, so
targeting the native `elementTemplates` section with `__ElementFromBinary`
instantiation is a possible future backend for the same transform — it would
remove even the skeleton's JS execution from the first frame. That backend is
tracked but gated on encode-pipeline and cross-platform support.

## Measured results

From the [strategy benchmark](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/REPORT.md) (~1000-element synthetic screens, `--jitless` interpreter, warm medians) — render time and the ops payload that would cross the thread boundary:

| scene | unlowered | lowered | ops payload |
|---|---|---|---|
| static-heavy (99.7% static) | 10.9 ms | **0.74 ms** | 77.6 KB → **69 B** |
| content feed (37% static) | 8.4 ms | **1.26 ms** | 60.4 KB → 9.2 KB |
| v-for list (12% static) | 6.6 ms | **1.44 ms** | 45.1 KB → 17.1 KB |

In a real browser (Lynx for Web) under 4× CPU throttling, `ifr+et` reached FCP −17% vs −12% for plain IFR on the hello-world screen — the lowering advantage grows as CPUs get slower and screens get bigger. See the [IFR page](/guide/ifr#measured-performance) for the full picture including costs.

## Semantics guarantee

Lowered and unlowered renders produce identical documents (verified in CI by
compiling every test template both ways and comparing the rendered output,
and by structurally comparing real example bundles built with the flag on and
off). The only observable difference is the order in which attributes were
set on nodes that mix baked-static and dynamic props — meaningless to
rendering.

One practical note: because interior nodes of a template are anonymous on
the main thread (no per-node bookkeeping), they don't carry `vue-ref-N`
selector attributes. Template roots keep them, and anything you can `ref`
is never an interior node, so NodesRef-based code is unaffected.
